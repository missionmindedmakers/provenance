# C2PA Core Concepts for Provenance and Attribution  
C2PA (Content Credentials) attaches a **cryptographically signed “manifest”** to an asset, recording who created/edited it, what was done, when, how, and what sources were used【14†L131-L139】. In C2PA terminology, the manifest (a.k.a. Content Credential) contains **assertions** — labeled statements such as actions, content bindings, or metadata. Each manifest is **hard-bound** to the exact asset (e.g. via a SHA‑256 hash of its bytes)【44†L105-L110】, so any tampering breaks the signature.  Standard assertions include **content bindings** (e.g. `c2pa.hash.data` hashes) and **actions** (e.g. `c2pa.actions` listing “created”, “edited”, etc)【44†L105-L110】. Every manifest **starts with a “created” or “opened” action** tied to an ingredient (the source asset)【32†L81-L85】【30†L163-L170】. For example, if you edit an existing document, the manifest’s `c2pa.actions` assertion would have an action like `c2pa.opened` pointing to an ingredient assertion for the original version (marked `parentOf`)【30†L163-L170】.

# Ingredients and Source Tracking  
Key to attribution is the **ingredient model**. Any external content (files or pasted text) is treated as an *ingredient* in the new document’s provenance. C2PA uses an **ingredient assertion** to list all source assets: e.g. one asset can be marked `parentOf` the current asset, or multiple assets marked `componentOf` it【25†L251-L259】. The guidance notes that *“ingredients are the key to the establishment of provenance”*, since each ingredient can itself carry its own C2PA manifest【25†L251-L259】【7†L2232-L2241】. In practice, when importing a file that already has C2PA data, the editor should copy that asset’s manifest into the new document’s manifest store (so the chain stays intact)【7†L2232-L2241】.  If a source has no manifest, you would still record it as an ingredient (e.g. by URL or title) and possibly add its author or citation. In all cases the new manifest should include an ingredient entry linking to each source. For example, the newspaper-editing scenario shows the editor’s manifest containing an `opened` action and an ingredient for the original photo marked “parentOf”【30†L163-L170】.

# Assertions for Metadata and Attribution  
Within the manifest you can include rich metadata about the content.  A common assertion is `stds.schema-org.CreativeWork`, which holds schema.org fields for the asset (title, author, license, etc.). For instance, a manifest might contain:  
```
"label": "stds.schema-org.CreativeWork",
"data": { "@context": "http://schema.org/", "@type": "CreativeWork",
  "author": [ { "@type": "Person", "name": "Gavin Peacock" } ] }
```  
This records the author of the content【11†L92-L100】. Other standard assertions include camera/EXIF info (`stds.exif`), IPTC metadata (`stds.iptc`), or any W3C Verifiable Credential for identity.  In short, use the **CreativeWork (or similar) assertions** to embed author/creator identity, title, citations, etc.  Each assertion (and even the entire claim) can include an optional `dataSource` field if you want to note where that information came from (e.g. user input vs embedded metadata)【47†L105-L114】.

# Actions and Claim Updates  
A C2PA **claim generator** (our editor) adds a new signed manifest each time significant changes occur【27†L171-L174】.  For example, on initial document creation or major export you issue a *standard manifest* with a hard binding to the file. If only the metadata changes (not the text itself), you’d use an *update manifest* to add or redact assertions. Typical workflow: 
- **Create** (new document): `c2pa.created` action, with any ingredients (sources) used.  
- **Open/Edit**: `c2pa.opened` action linking to the parent version, then actions like `c2pa.edited` or even custom labels for copy/paste.  
- **Paste/Import**: treat as adding content – you might record a specific edit action, but more importantly add those sources as ingredients. (If the pasted content had a manifest, fetch and merge it; otherwise record its provenance fields manually.)  
- **Save/Export**: produce a new manifest signed by the editor. Each claim is signed (often with a timestamp) so consumers can verify the chain.  

C2PA enforces that the first action in every manifest be either `created` or `opened` and that it reference an ingredient【32†L81-L85】.  In practice our editor would act as the “claim generator” and append a new manifest on each export; if anything is removed or changed from previous manifests, those assertions are simply redacted in the new one【27†L171-L174】. For example, Cloudflare’s image service shows how two manifests coexist: one by the camera (`c2pa.captured`) and one by Cloudflare (`c2pa.resized`), each signed separately【43†L242-L250】. 

# Content Binding and Integrity  
Every manifest includes content bindings (hashes) to lock it to the document text. The C2PA default is a SHA-256 binding. In practice, the tool would hash the entire text (or byte ranges) and include a `c2pa.hash.data` assertion【44†L105-L110】.  That way, if someone later edits the text without updating the manifest, verification will fail. The C2PA SDKs typically handle these bindings automatically, but as a designer you’d ensure to include them. This “hard binding” is mandatory: every standard manifest must contain a binding to the asset【27†L171-L174】【44†L105-L110】.

# Storage and Embedding of C2PA Data  
Since a Notion-style document (text or HTML) usually cannot embed JUMBF boxes, C2PA data is typically stored **externally**. The C2PA spec allows a **manifest repository**: all the manifests for the asset are placed in a linked store【25†L198-L202】. For example, the editor might save manifests to a cloud service or database, and reference them via HTTP Link headers or a known URL associated with the document ID【25†L198-L202】. This way the asset remains lightweight but its provenance is fetchable. If a format *does* allow embedding (e.g. PDF with metadata), you could embed the C2PA Manifest Store directly; otherwise plan an external store. In either case, the active manifest (latest claim) is always included in the store (with older ones for history).  

# Summary of an Attribution Workflow  
- **Initial save/export**: Editor creates a new C2PA manifest. It includes a `created` action, content binding hashes, and metadata assertions (title, author=editor, etc.). Any external files or copied text are listed as ingredients (with their references or manifests)【7†L2232-L2241】【25†L251-L259】.  
- **Adding content**: When the user pastes text or inserts a file, treat that event as adding ingredients. If the source file had a C2PA manifest, fetch and merge it【7†L2232-L2241】. Add a new action (e.g. a custom “pasted” or just an `edited` action) and an ingredient assertion pointing to the source (marked `componentOf`). Include any relevant metadata (e.g. original author from the source) in the new manifest.  
- **Subsequent edits**: On each publish or export, append a new manifest. Use `c2pa.opened` if editing a previous version, and `c2pa.actions` for each edit. The editor (claim generator) signs the new claim【27†L171-L174】. All previous asserts carry over unless redacted.  
- **Verification**: Consumers (or the editor itself) can run a C2PA validator to fetch the manifest store, verify each signature, and read out the provenance chain. They’ll see who authored each part (from `CreativeWork` assertions) and what sources were used (from ingredient assertions). 

By combining content hashes, signed assertions, and ingredient tracking, C2PA provides a detailed, tamper-evident record of **who wrote or contributed each piece of content**. For example, the manifest might show “Author: Alice” for original paragraphs, then “Ingredient: external_doc.pdf” for copied text, each with its own metadata. This ensures that all pasted or imported content is attributed to its origin when the document’s provenance is inspected【11†L92-L100】【25†L251-L259】.
